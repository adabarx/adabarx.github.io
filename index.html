Keyboard Generator

Condition: Current priority. First time doing a lot of concurrent processing, very interesting problems.

I don't like the QWERTY layout, it hurts my wrists. I could use Dvorak or a Coleman variant, but there's one major problem: I use vim. Specifically, I need the HJKL keys to stay in the same place. So I created a program that uses a genetic algorithm: randomly generating/modifying keyboards and running them in a fitness function that uses all my repos as training data.

But there's a problem, local optima, I don't know if the generated keyboard is the best keyboard or just a good keyboard. The solution? Throw more compute at it! Generate 100 good keyboards and one of them is bound to be the most optimal one. But even when parallelizing with Rayon, it's a lot of work for my laptop.

The time complexity of the fitness function is inherently linear to the training data used. So the only way I can speed up the keyboard generation is by massively parallelizing the process. I would like to explore if there is any way I could utilize the GPU. But there are issues with that. So for now I'm implementing it as a REST API and building a separate composer app (now using Go!) so I can distribute the workload among all the old laptops, NUCs, and raspberry pis that I've accumulated over the years.

Float Crush

Condition: Core process and parameters are feature complete. Needs a decent GUI.

My first real audio effect! It started when I was trying to figure out why DAWs process samples with 32 bit floating point numbers instead of integers. After a lot of reading about IEEE 754, I ended up wanting to make a bitcrusher that could crush audio into any arbitrary floating point size. Using a two-stage modified binary search for quantization, I ended up creating a process where I could parameterize the size of the exponent and mantissa of the sample being processed, emulating the sound of audio that's been recorded to a lower floating point size.

Because I'm emulating lower bitrates with a search instead of trying to use efficient bitwise math, I'm able to parameterize a lot of fun aspects of the processing. Leading to a lot of fun, noisey sounds. I'm proud of this project. I have plans to expand it, including a good GUI, but I'm happy letting it rest for right now.

Katlang

Condition: Abandoned, only got to step 2 of the ‘build a compiler' process. Coming back to the idea at a later point in time.

I believe that within Rust, there is a much smaller and cleaner language struggling to get out. This is my testing ground to try and find it. Right now, it just parses a source file into an AST. I eventually want to rewrite it all in Zig and have the compiler output Zig code. The idea is to make a language that has a similar feel to safe Rust. But instead of using unsafe Rust, you would switch to writing Zig.

ModulaDAW

Condition: Abandoned. Very buggy proof of concept. Coming back to the idea at a later point in time.

I would love to build DAW one day and this is the first whiteboard sketch of that idea. It's just a node graph where each node would be an effect and the graph connections would represent the signal paths. There are a lot of apps like this already, but they're annoying to use because the connecting paths look like a mess. So the hook for this is that I would develop a GPU-based path finding algorithm that the connections would use to self-organize and reduce visual clutter. It's very buggy and I'm going to rebuild it at some point once I've learned more.
